01

The journey of a click

You write a tiny JSON file describing a browser flow. Out comes a 30-second vertical video with narration and captions. Here's what happens in between.

what it is

A pipeline that turns scripts into scroll-stopping demo videos

Clipwright is a tool for making short-form how-to videos. You tell it what browser actions to perform — go here, click this, type that — and it records the session, trims the boring pauses, adds a human-sounding voiceover, stamps word-synced captions on the screen, and renders a vertical MP4 ready for TikTok, Reels, or Shorts.

There's a CLI (you run clipwright record, clipwright render, etc. from the terminal) and a desktop app that wraps it in a clickable UI.

zooming in

Imagine you run this one command

You've just edited browse-plan.json to describe a demo of adding a book to a library. You hit enter. Here's what actually happens over the next ~60 seconds:

1
A real browser launches silently

Playwright opens Chrome at your target URL. A video recording starts.

2
Clicks, types, and scrolls play out

Every action from your plan runs. Each one gets stamped with a precise timestamp inside moments.json.

3
Dead time gets trimmed

Silent pauses between meaningful actions are collapsed into tight segments.

4
A script skeleton is drafted

One empty narration clip is generated per segment, with timing hints. You (or an AI) fill in the words.

5
The voiceover is synthesized

A TTS model speaks the script and returns audio plus character-level timing data.

6
Captions are rendered as PNGs

Two-word chunks become transparent images, each tagged with the exact millisecond it should appear.

7
ffmpeg stitches it all together

The vertical frame, voiceover, captions, and branded outro are composed into final.mp4.

💡
This is a pipeline

Each step takes a file, does one thing to it, and writes a new file. The next step picks up where the last one left off. Engineers call this pattern a pipeline — and it's everywhere in software, from video editing to data processing to continuous integration.

what you write vs. what you get

The magic isn't in the plan — it's in what surrounds it

Your input is tiny. Your output is a production-quality video. Look at how little you write:

code · browse-plan.json 
"actions": [
  {"type": "navigate", "label": "Open discover",
   "chapter": "library",
   "fields": {"url": "/discover"}, "wait": 3.0},
  {"type": "type", "label": "Search Blue Lock",
   "chapter": "library",
   "fields": {"selector": "input[placeholder*='search' i]",
              "text": "blue lock"},
   "wait": 2.5},
  {"type": "click", "label": "Add to library",
   "chapter": "library",
   "fields": {"selector": "button:has-text('ADD TO LIBRARY')"},
   "wait": 4.0}
]
plain english

Action 1: Go to the /discover page. Wait 3 seconds for it to settle.

Action 2: Find the search box (any input whose placeholder contains "search"). Type "blue lock". Wait 2.5 seconds.

Action 3: Click the button with the text "ADD TO LIBRARY". Wait 4 seconds — long enough that the animation completes on screen.

The label on each action becomes a hint the script-writer (AI or human) uses when composing the narration. The chapter groups actions into one narrated segment.

🎯
Why should you care?

The payoff of understanding this structure: when an AI agent is writing a plan for you and it gets a field wrong, you'll be able to spot the problem in 2 seconds. "That wait is way too short for a page load" is the kind of intuition that separates good AI steering from bad.

02

Meet the crew

Clipwright is a film production crew in code form. Each role does one thing well, and they hand work off to each other in a fixed order.

the cast

Seven specialists, one film

Think of a small film crew: each person owns a craft. The director plans the shoot, the camera operator captures it, the editor cuts it, the composer scores it, the subtitler times the words, and the colorist finishes the look. Clipwright works exactly the same way.

D
The Director — browse-plan.json

A declarative list of actions that the shoot must follow. No code, just intent.

C
The Camera Operator — Playwright Recorder

Drives a real Chrome browser through the plan and captures a continuous video.

E
The Editor — Segments & Keyframes Planner

Looks at the timestamps, trims the dead pauses, and decides where each scene begins and ends.

V
The Voice Actor — TTS provider

Speaks the written script. Returns not just audio but exactly when each character is spoken.

S
The Subtitler — Caption Chunker + PNG Renderer

Splits the narration into 2-word UPPERCASE chunks and paints each one as a transparent image.

F
The Finisher — ffmpeg Composer

Assembles every clip, lays the captions on top, splices the outro at the end, and exports final.mp4.

B
The Brand Designer — Outro Renderer

Paints a branded end-card with the project's name and URL using PIL.

what this looks like in the code

Open the src/clipwright/ directory and you'll see the whole cast, each in their own dressing room:

📦
This is called "separation of concerns"

Each folder has exactly one job. If the voice provider breaks, you only have to look in tts/. If the captions are off, you look in captions/. You never have to wade through 5,000-line files. When you ask an AI to "fix the caption timing," it knows exactly which folder to open.

the producer's script

One function calls the whole crew

The CLI (cli.py) is the producer: it doesn't do any craft itself, it just tells each specialist when to work. Here's the heart of the render stage — the finisher assembling the film:

code · render/composer.py 
"""The render pipeline:
per-segment extract -> concat -> subtitle overlay LAST.

Hard rules honored:
  1. Subtitles applied last in the filter chain.
  2. Per-segment extract then lossless concat;
     overlays on top of the concat product.
  3. 30ms audio fades at every boundary
     (applied in the compose filter).
  4. Overlays use PTS-shifted setpts
     via per-window enable='between(...)'.
"""
plain english

This is a module docstring — the comment at the top of a file that explains what lives inside.

Rule 1: Captions go on after everything else is composed. (If you put them on first, other layers hide them.)

Rule 2: Cut each segment separately, then glue them together without re-encoding. (Re-encoding twice destroys quality.)

Rule 3: Fade audio in and out at every segment seam — only 30 milliseconds, enough to prevent an audible "pop" when clips are joined.

Rule 4: Caption images are told exactly when to appear and disappear, relative to the composed timeline.

These rules aren't style — they're production correctness. Break any of them and the final video silently looks or sounds broken.

🎯
Why should you care?

When an AI is rewriting ffmpeg code for you, these four rules are the exact things it will silently violate. Knowing they exist — and what they look like when broken (pops, hidden captions, grainy output) — is how you catch the mistake before it ships.

03

Passing the baton

How does the camera operator tell the editor "here's where the clicks happened"? Through a relay race of JSON files — each stage writes the baton the next stage picks up.

the metaphor

Relay runners, not conversations

In a relay race, runners never yell instructions to each other — they just pass a baton at a predetermined spot. The next runner grabs it and goes. That's how Clipwright's stages coordinate: nobody calls anybody else. They just drop a JSON file in a known location, and the next stage picks it up.

This is called file-based pipelining. It's old-fashioned, unglamorous, and absolutely bulletproof — if the pipeline crashes halfway, you can pick up exactly where it stopped by reading the last-written file.

💡
Files are the contract

No stage knows how the others are implemented. The recorder doesn't care if the editor is written in Python, Rust, or Klingon — it just writes moments.json in a known shape. Swapping out any stage is easy as long as the new version still reads and writes the same files.

sampling a baton

The recorder's log, decoded

When the camera operator finishes, the baton it hands off is called moments.json. Here's the code that stamps each moment:

code · record/playwright_recorder.py 
class _MarkRecorder:
    """Async callable collecting moments relative to recording start."""

    def __init__(self) -> None:
        self._t0: float | None = None
        self.moments: list[Moment] = []

    def start(self) -> None:
        self._t0 = time.monotonic()

    async def __call__(self, action: str, **data: Any) -> None:
        if self._t0 is None:
            return
        t = time.monotonic() - self._t0
        self.moments.append(Moment(t=t, action=action, data=data))
plain english

This is a little class called _MarkRecorder. It's one object with a job: remember the moment every action happens.

When it's born, it has no start time yet and an empty list.

start() notes the exact moment the recording began, using a clock that never goes backwards — time.monotonic(). (Normal clocks can jump if your computer's time gets corrected; monotonic ones can't.)

Every time an action fires, this object is called like a function. It calculates how many seconds since start, and appends a new Moment to its list.

After the shoot ends, that list gets saved to moments.json. That's the baton.

04

Hired specialists

Clipwright doesn't try to do everything itself. It hires three specialist services — a browser driver, a voice studio, and a video-composition house — and orchestrates them.

the metaphor

A small film company hiring subcontractors

A small studio doesn't own its own soundstage, voice-recording booth, and compositing suite. They hire specialists. Clipwright is the same: it's the producer calling the shots, but the actual heavy lifting is done by three mature, battle-tested tools that already exist.

🌐

Playwright

A browser automation tool by Microsoft. It can drive Chrome, Firefox, or Safari programmatically — clicking, typing, and capturing video exactly like a human would.

🎙️

TTS Providers

Three pluggable voice engines — Kokoro (free, near-human), Piper (tiny, offline), or ElevenLabs (paid API, highest quality). Same contract, different trade-offs.

🎬

ffmpeg

The world's most powerful open-source video tool. Can cut, stitch, re-encode, overlay, and blur video in one command. Clipwright calls it like a command-line assistant.

🎯
Why should you care?

Understanding which work is done by a third-party tool vs. Clipwright's own code tells you where to look when something goes wrong. A 403 error from the TTS? Check your API key. A weird video artifact? It's probably ffmpeg. A button that wouldn't click? Playwright couldn't find it.

hiring the voice studio

How Clipwright calls ElevenLabs

Here's the actual code that reaches out to ElevenLabs' servers to generate narration. Notice it doesn't use any SDK — just a plain HTTP request. Keep it simple:

code · tts/elevenlabs.py 
    body = {
        "text": text,
        "model_id": model_id,
        "output_format": output_format,
        "voice_settings": {
            "stability": stability,
            "similarity_boost": similarity_boost,
            "style": style,
        },
    }
    req = urllib.request.Request(
        ENDPOINT.format(voice_id=voice_id),
        data=json.dumps(body).encode("utf-8"),
        headers={
            "xi-api-key": key,
            "Content-Type": "application/json",
            "Accept": "application/json",
        },
        method="POST",
    )
plain english

Build a package of settings: the text we want spoken, which voice model, what audio format, and how stable/expressive the voice should be.

Wrap it in a web request. The xi-api-key header is how ElevenLabs knows who is asking — it's your identity badge.

POST means "I'm sending data" (as opposed to GET, which means "I'm just asking for something"). The URL includes the exact voice ID we picked.

Because this is a paid service, every request like this costs real money. That's why the pipeline caches the audio file locally — you never pay twice for the same narration.

swappable specialists

Same job, three different subcontractors

Each TTS provider is a separate file in tts/. All three speak different languages internally, but they all hand back the same thing: an audio file plus timing data.

Kokoro · default

Free, local, near-human

Open-source model that runs on your own machine. Downloads a ~2 GB model once, then costs nothing per use. Great default.

Piper · offline

Tiny, offline, natural

Ultra-compact. Doesn't emit native word timings, so Clipwright force-aligns the audio with a mini transcription model.

ElevenLabs · paid

API, highest quality

Commercial service. Needs an ELEVENLABS_API_KEY. Character-level timestamps are built into the response.

🔌
This is called the "strategy pattern"

When multiple classes can do the same job but in different ways, and a shared interface lets you swap between them without rewriting the rest of the system — that's the strategy pattern. Ask an AI to "add a new strategy for X" and they'll know exactly what you mean.

05

The darkroom rules

Developing a photo in a darkroom is a fixed sequence — reverse the order and the print is ruined. Rendering video is the same. Here are Clipwright's non-negotiable rules, and why each one matters.

rule 1

Subtitles go on last, not first

The tempting way

Burn captions into each clip

Render the captions on every segment first, then concatenate. Seems efficient — one pass per segment.

The correct way

Overlay captions last, on the whole film

Concatenate clean segments first. Then paint captions over the composed video as the very last step.

Why it matters: any overlay you add after captions — a logo, a transition, even a blurred backdrop — will hide the captions. Making captions the last layer guarantees nothing ever covers them.

⚠️
This is a silent failure

Breaking this rule doesn't cause a crash — it produces a video that looks almost right but has captions slightly hidden or misaligned. The scariest bugs are the ones where nothing seems wrong.

rule 2

Cut each segment, then stitch — never re-encode twice

When you edit video, every time you "re-encode" the file (convert it from one format to another), you lose a little quality. It's like photocopying a photocopy.

Overlay captions once on the final

Clipwright encodes exactly once: when captions go on the composed film. Everything before that is lossless copy-and-paste.

rule 3

30 milliseconds of fade prevents an audible pop

When you glue two audio clips together, the waveform at the seam jumps abruptly. Your ear hears that jump as a tiny click. The fix is comically simple: fade the audio in and out by 30 ms at every boundary.

code · render/compose_vertical.py 
def audio_filter(
    *,
    lead: float,
    duration: float,
    src_label: str = "[1:a]",
    out_label: str = "[aout]",
    edge_fade: float = 0.03,
) -> str:
    """Pad audio with lead silence, fit total duration,
    30ms fades per Hard Rule 3."""
    delay_ms = int(lead * 1000)
    fade_out_start = max(0.0, duration - edge_fade)
    return (
        f"{src_label}adelay={delay_ms}|{delay_ms},"
        f"apad=whole_dur={duration},atrim=duration={duration},"
        f"afade=t=in:st=0:d={edge_fade},"
        f"afade=t=out:st={fade_out_start:.3f}:d={edge_fade}{out_label}"
    )
plain english

This function builds a filtergraph string that ffmpeg will run on the audio.

First: add a tiny silence at the beginning (adelay) so the narration doesn't start instantly.

Second: pad or trim to the target segment duration (apad/atrim) so the voiceover fits exactly in its slot.

Third: fade the audio in for 30 ms at the start, and fade out for 30 ms at the end. That's the pop-killer.

You can't hear a 30 ms fade. You can hear what happens without it. That asymmetry is why this rule exists.

rule 4

Stretch audio without changing pitch

Sometimes the TTS returns audio that's slightly too long for the segment it's meant to fill. You need to make it shorter — but if you just play it faster, the voice turns chipmunky.

ffmpeg has a built-in for exactly this: atempo. It changes the speed without touching pitch. But there's a catch: each atempo can only go from 0.5× to 2.0×. For wider stretches, you chain them.

code · ffmpeg.py 
def atempo_chain(ratio: float) -> str:
    """Build an ffmpeg filtergraph string stretching audio by `ratio`.

    ffmpeg's `atempo` accepts 0.5..2.0; chain multiple for wider ratios.
    ratio > 1 speeds up; ratio < 1 slows down. Pitch is preserved.
    """
    if ratio <= 0:
        raise ValueError(f"ratio must be positive, got {ratio}")
    parts: list[str] = []
    r = ratio
    while r > 2.0:
        parts.append("atempo=2.0")
        r /= 2.0
    while r < 0.5:
        parts.append("atempo=0.5")
        r /= 0.5
plain english

This function stretches audio by some ratio. Ratio 1.2 = 20% faster. Ratio 0.8 = 20% slower.

Because ffmpeg's single-pass tempo change can only handle 0.5× to 2.0×, the function chains multiple passes if the user wants something extreme (like 3×).

The while-loops peel off "maximum" stretches until the remaining ratio fits in the allowed range. 3× becomes 2× then 1.5×. 0.3× becomes 0.5× then 0.6×.

A tiny utility that papers over an annoying library limitation. This is exactly the kind of code you want AI to produce for you — and to spot when it gets it wrong.

💡
Rule of thumb

Clipwright never slows narration down — only speeds it up if the audio is a bit too long. The reasoning: a short voiceover with trailing silence feels natural. A slowed-down voice sounds drugged.

The debugging superpower

the checkpoint habit

Always look at the intermediate file first

Because every stage writes a file, you can see exactly what went wrong before it cascades. Is the final video bad? Open the previous stage's file first.

From one set to a studio lot

The bigger picture: a pluggable pipeline wrapped in a desktop app, wired into an AI agent, and designed from day one to grow from one video to many.

the architecture

Three layers, one mission

The desktop app doesn't reimplement the pipeline. It shells out to the Python CLI using Tauri, watches the output files, and reflects what it sees. The agent surface is similarly thin — a document that teaches Claude Code how to drive the CLI. Layer 1 stays authoritative. Layers 2 and 3 are just different skins.

designing for growth

One project, many videos

A real user making videos about an app wants multiple clips from the same project — "signup flow," "reading a chapter," "settings tour." Each one shares the brand, voice, and site, but has its own plan, script, and final.

The layout was recently redesigned to reflect this — notice how the outer config stays shared while everything video-specific gets its own folder:

.clipwright.jsonshared defaults — name, URL, aspect, voice, brand
videos/every video lives inside here
signup-flow/one clip
video.jsonper-video overrides (aspect, template, notes)
browse-plan.jsonthis video's own action list
script.jsonthis video's narration
out/this video's artifacts & final.mp4
reading-chapter/another clip, same brand
reader-settings/another clip, same brand

Any field in video.json overrides the project-level default. Missing? Falls back to the shared config. This is called configuration layering — it's the same pattern you see in VS Code (user settings → workspace settings) and Git (global → repo → local).

the override pattern, in code

A tiny config class that holds only the differences

code · config.py 
@dataclass
class VideoConfig:
    """Per-video overrides. Any None/empty field falls back to ProjectConfig."""
    slug: str = "main"
    title: str = ""
    aspect: Optional[str] = None
    target_duration: Optional[float] = None
    template: Optional[str] = None
    notes: str = ""
    extra: dict = field(default_factory=dict)
plain english

This is a small data container that describes a single video inside a project.

Most of its fields are optional (that's what Optional[...] means). If a video doesn't declare its own aspect, the pipeline falls back to the project-level default.

The slug is the folder name — short, URL-safe. The title is what appears in the UI.

The extra dict is an escape hatch: anything not pre-declared (like new fields you add later) lands there without breaking older projects.

This compact shape is powerful: one project can hold dozens of videos, and each only stores what makes it different.
